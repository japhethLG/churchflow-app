# PWA & Android TWA

How ChurchFlow ships as an installable Progressive Web App and is packaged
into an Android APK (Trusted Web Activity) for the Play Store.

> **Audience:** anyone touching the service worker, the web manifest, the
> Firebase auth flow, or the Android packaging. The non-obvious parts are the
> Serwist-on-Turbopack setup (§2) and the TWA login fix (§5) — read those
> before changing auth or the SW.

---

## 1. Overview

```
Next.js app (web)  ──manifest + service worker──►  installable PWA
       │
       │  PWABuilder wraps the deployed PWA
       ▼
Android TWA (APK)  ──Digital Asset Links──►  fullscreen, no URL bar
```

- **Web app manifest** ([src/app/manifest.ts](../src/app/manifest.ts)) — makes
  the site installable; `display: standalone` removes browser chrome.
- **Service worker** (Serwist) — precaches the app shell, serves an offline
  fallback, and runtime-caches assets. Critically, it passes Firebase Auth
  traffic straight through (§2.1).
- **Android TWA** — the APK was generated by **PWABuilder** (not Bubblewrap;
  see §6). It loads the live site fullscreen once Digital Asset Links verify
  the relationship between the APK signing key and the domain (§4).

The live site is `https://churchflow.crabdance.com`; the API is
`https://api.churchflow.crabdance.com`.

## 2. Service worker (Serwist on Turbopack)

This app uses **`@serwist/turbopack`**, NOT `@serwist/next`. `@serwist/next` is
webpack-only and breaks under Next 16's Turbopack. The Turbopack variant serves
the worker through a **Route Handler** instead of a webpack plugin.

| File | Role |
|---|---|
| [src/app/sw.ts](../src/app/sw.ts) | The service worker source — caching strategies, offline fallback, auth passthrough. |
| [src/app/serwist/[path]/route.ts](../src/app/serwist/%5Bpath%5D/route.ts) | `createSerwistRoute(...)` compiles `sw.ts` with esbuild and serves it at `/serwist/sw.js`. Dynamic `[path]` segment because the route helper requires a path param. |
| [src/app/~offline/page.tsx](../src/app/~offline/page.tsx) | Offline fallback shown for document requests when the network is unavailable. Precached with a git-SHA revision so deploys bust it. |
| [next.config.ts](../next.config.ts) | `withSerwist(...)` wraps the config (adds esbuild to `serverExternalPackages`). |
| [src/app/layout.tsx](../src/app/layout.tsx) | `<SerwistProvider>` registers the SW — **production only** (§2.3). |

`useNativeEsbuild: true` in the route handler avoids bundling esbuild's native
binary into the server build. The precache `revision` is `git rev-parse HEAD`
(falls back to a random UUID), so each deploy invalidates stale precached
responses.

### 2.3 Registration is production-only — never run the SW in dev

`<SerwistProvider>` in [layout.tsx](../src/app/layout.tsx) is gated with
`disable={process.env.NODE_ENV !== "production"}`, so `npm run dev` never
registers the worker. Test the SW with a production build (`npm run build &&
npm run start`).

Why this matters: a service worker is scoped per **origin**, not per port or
build config. If a *production*-config SW ever gets registered on `localhost`
(e.g. running `npm run start` locally to test the APK), it keeps controlling
`localhost` afterward — serving a stale precached app shell and cached
cross-origin API responses (`defaultCache` uses `NetworkFirst` with a 1h TTL
for cross-origin in production). The result is the dev app silently failing to
talk to the backend even though prod is fine.

`disable` only **skips registration** — it does NOT unregister an already
installed SW. If you hit the stale-SW symptom, clear it once by hand:
DevTools → Application → Service Workers → **Unregister**, then **Clear site
data**, then hard reload (or use `chrome://serviceworker-internals`).

### 2.1 Auth passthrough — the SW must NOT touch Firebase Auth

The first runtime-caching rule in [sw.ts](../src/app/sw.ts) is `authPassthrough`
(`NetworkOnly`), listed **before** `defaultCache` so it wins (first match wins):

```ts
const authPassthrough = {
  matcher: ({ url }) =>
    /(^|\.)(googleapis\.com|firebaseapp\.com|google\.com|gstatic\.com|firebaseio\.com)$/.test(url.hostname)
    || url.pathname.startsWith("/__/"),
  handler: new NetworkOnly(),
};
```

It matches both the Google/Firebase auth hosts **and** our self-hosted auth
handler under `/__/` (§5). Caching or mishandling these requests breaks
`signInWithRedirect`. `defaultCache` already excludes `/api/` so Bearer- and
cookie-gated responses never go stale (a stale 401 would trigger a global
sign-out — see [proxy.ts](../src/proxy.ts) + [client.ts](../src/lib/api/client.ts)).

### 2.2 Two different CSPs — don't confuse them

There are **two** Content-Security-Policy headers in play, and the TWA login
debugging hinged on telling them apart:

| CSP | Set in | Governs |
|---|---|---|
| **Page CSP** (nonce + `strict-dynamic`) | [proxy.ts](../src/proxy.ts), per-request | What the *page* may load/connect to. |
| **Service-worker CSP** | [next.config.ts](../next.config.ts) `serviceWorkerHeaders` on `/serwist/sw.js` | What the *service worker's own fetches* may reach. |

The SW CSP is `default-src 'self'; connect-src 'self' https:; script-src 'self'`.
The `connect-src https:` is load-bearing: the SW fetches cross-origin for
runtime caching and passes Firebase Auth through. Without it the SW can only
fetch same-origin and Google auth calls fail with `auth/network-request-failed`
— even though the page CSP allows them, because the SW intercepts the request
first.

The page CSP carve-outs for auth (in [proxy.ts](../src/proxy.ts)):
- `connect-src` includes `https://*.firebaseapp.com https://apis.google.com
  https://accounts.google.com` plus identitytoolkit/securetoken/googleapis.
- `frame-src` includes `accounts.google.com` + `*.firebaseapp.com`.
- `worker-src 'self'` and `manifest-src 'self'` added for the SW + manifest.

## 3. Web manifest & install metadata

[src/app/manifest.ts](../src/app/manifest.ts): `name`/`short_name` "ChurchFlow",
`display: standalone`, `orientation: portrait`, `start_url: "/"`, `scope: "/"`.
Icons live in [public/icons/](../public/icons/) — 192, 512 (`purpose: any`) and a
512 maskable. The maskable icon matters for Android's adaptive icon mask.

The proxy matcher and CSP both exclude `manifest.webmanifest`, `icons`, and
`serwist` so they're served without a nonce or auth gate.

## 4. Digital Asset Links (Android TWA)

For the TWA to run fullscreen (no URL bar), Android verifies that the APK
signing key is authorized for the domain via
[public/.well-known/assetlinks.json](../public/.well-known/assetlinks.json):

```json
[{ "relation": ["delegate_permission/common.handle_all_urls"],
   "target": { "namespace": "android_app",
               "package_name": "com.crabdance.churchflow.twa",
               "sha256_cert_fingerprints": ["A7:EC:69:..."] } }]
```

- `package_name` = `com.crabdance.churchflow.twa`.
- The SHA-256 fingerprint is **PWABuilder's signing key**. If you switch to
  Google Play App Signing, add Google's re-signed cert fingerprint too (the
  array can hold multiple).
- **`/.well-known` is in `PUBLIC_PATHS`** in [proxy.ts](../src/proxy.ts). Android's
  verifier fetches it with no cookie; it must never redirect to `/login`. Note
  the dot-prefixed path is NOT caught by Next's matcher exclusion — it's the
  `PUBLIC_PATHS` entry that does the work.

## 5. Login inside the TWA — the dual sign-in path

`signInWithPopup` cannot complete inside a Trusted Web Activity (the popup can't
message its credential back through the Custom Tab sandbox). So sign-in branches
on whether the app is running standalone — see
[src/lib/auth/actions.ts](../src/lib/auth/actions.ts):

```
isStandaloneLaunch()  ─true─►  signInWithRedirect  ─►  (page navigates away)
                      ─false─► signInWithPopup     ─►  completeSignIn()
```

`isStandaloneLaunch()` checks `display-mode: standalone`, iOS
`navigator.standalone`, and an `android-app://` referrer. The redirect path
resolves `null` (the page is already navigating); the credential is finalized by
`completeRedirectSignIn()`, called on mount of any sign-in-initiating page
([LoginButton.tsx](../src/app/(auth)/login/LoginButton.tsx),
[InviteTokenPage.tsx](../src/components/pages/invite/InviteTokenPage.tsx)).
`completeSignIn(user)` is the shared tail: backend session exchange → force
token refresh → mint Next cookie.

### 5.1 Self-hosted auth handler (the storage-partitioning fix)

After the OAuth round-trip, `getRedirectResult()` returned nothing inside the
APK and the user landed back on `/login` silently. Cause: Chrome **partitions
third-party storage**, so when Firebase's auth handler runs on the cross-origin
`*.firebaseapp.com`, the redirect result can't be read back on our origin.

Fix: **self-host the Firebase auth handler** so it's first-party. The SDK is
pointed at our own domain and we proxy the handler back to Firebase:

1. **`NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN = churchflow.crabdance.com`** (was
   `church-test-f4537.firebaseapp.com`). Now the SDK hits `/__/auth/*` on our
   origin.
2. **Rewrites** in [next.config.ts](../next.config.ts) proxy `/__/auth/:path*`
   and `/__/firebase/:path*` to `${NEXT_PUBLIC_FIREBASE_PROJECT_ID}.firebaseapp.com`.
3. **[proxy.ts](../src/proxy.ts) matcher** excludes `__/` so middleware doesn't
   gate the handler.
4. **[sw.ts](../src/app/sw.ts) `authPassthrough`** also matches `/__/` so the SW
   never caches it.

Because the handler is now served from our own origin, storage is first-party
and `getRedirectResult()` can read the credential.

> **Manual step (cannot be automated, must be done in Google Cloud Console):**
> add to the OAuth client used by this Firebase project:
> - Authorized JavaScript origins: `https://churchflow.crabdance.com`
> - Authorized redirect URIs: `https://churchflow.crabdance.com/__/auth/handler`
>
> Without this, Google rejects the redirect. After deploying, clear the app's
> storage on-device before re-testing.

## 6. Android packaging — PWABuilder (not Bubblewrap)

The shipped APK was generated by **PWABuilder** (point it at the live URL, it
emits a signed APK + `assetlinks.json` fingerprint). Bubblewrap was tried first
but its bundled JDK is x86-64 and the build host is aarch64 ("Exec format
error"); the leftover `church-app-twa/` Bubblewrap project is superseded and
unused.

To rebuild the APK: run PWABuilder against `https://churchflow.crabdance.com`,
keep the signing key consistent (or update `assetlinks.json` with the new
fingerprint), and upload to the Play Console.

## 7. Standalone-only UX tweaks

[src/app/globals.css](../src/app/globals.css) has an
`@media all and (display-mode: standalone)` block that only applies in the
installed app (not the browser tab):

- `html { touch-action: pan-x pan-y }` — kills pinch-zoom and double-tap zoom
  while leaving panning/scrolling intact.
- `body { -webkit-touch-callout: none; user-select: none }` — suppresses the
  long-press highlight + selection/search callout for a native feel.
- Re-enables `user-select: text` on `input`, `textarea`, and
  `[contenteditable="true"]` so typing/editing still works.

## 8. Gotchas

| Trap | Reality |
|---|---|
| Using `@serwist/next` | Webpack-only; breaks under Turbopack. Use `@serwist/turbopack` + the Route Handler. |
| Dev app can't reach the backend (prod is fine) | A stale production-config SW is controlling `localhost`. Registration is now prod-only (§2.3); unregister the leftover SW once in DevTools. |
| Adding Firebase auth hosts only to the page CSP | The SW intercepts first — its own CSP (`serviceWorkerHeaders`) must allow `connect-src https:` too. |
| Reverting `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN` to `*.firebaseapp.com` | Reintroduces the storage-partitioning bug; TWA login silently fails. |
| Caching `/__/auth/*` or Firebase hosts in the SW | Breaks `signInWithRedirect`. They're `NetworkOnly` via `authPassthrough`. |
| Expecting `/.well-known` to be excluded by the proxy matcher | Dot-paths aren't; it's covered by `PUBLIC_PATHS`. |
| Forgetting the GCP Console redirect URI/origin | Self-hosted auth handler 400s without `…/__/auth/handler` authorized. |
| `start_url`/`scope` mismatch | Both are `/`; a stale `/launch` start_url left the install pointing at a route that redirects. |
